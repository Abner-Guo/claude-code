# Design 阶段规范（V1.0）

## 1. 目标
1. 把 Requirement 转成可编码、可验证、可回滚的实施方案。
2. 在编码前冻结业务口径与接口契约，降低联调返工。

## 2. 输入
1. 已通过 Gate 的 `requirement.spec.md`。
2. 历史主设计：`docs/features/<feature>/design/design.md`（若存在）。
   - **强制要求**：主设计必须包含现有接口契约（请求/响应字段级结构），作为契约变更的基线。
   - 若主设计缺少接口契约，必须先补充后再进入 Design 阶段，否则 AI 无法生成完整的变更后契约样例。
3. 相关知识基座与代码证据。
4. `uncertainty/register.md` 与 `risk/register.md`。

## 3. 输出（强制）
1. 迭代设计文档：
   - `docs/features/<feature>/design/iterations/<date-topic>/design.md`
   - 必须包含变更后的完整接口契约（每个变更接口独立展示完整请求/响应字段，不抽公共部分）
2. 更新寄存：
   - `docs/features/<feature>/uncertainty/register.md`
   - `docs/features/<feature>/risk/register.md`
3. 评审通过后回写主设计：
   - `docs/features/<feature>/design/design.md`
   - 回写时同步更新主设计中的接口契约基线

## 4. 执行步骤
1. 影响面建模
   - 明确影响 controller/service/mapper/table 与兼容面。
2. 方案分叉与收敛
   - 至少 2 个方案，明确推荐方案与不选理由。
3. 契约矩阵输出
   - 明确接口级入参/出参与兼容策略。
4. 逻辑细化（可编码级）
   - 参数归一化规则。
   - 场景分流规则。
   - 查询过滤与聚合口径。
   - 返回结构与 `null/0` 语义。
5. 实施切片
   - 拆分为可独立交付、可独立验证的切片。
6. 人工 TODO 登记
   - 需要人工补充的信息登记 `TODO-DES-*` 并绑定 owner。

## 5. Design 文档最小结构
1. 文档信息。
2. 设计目标。
3. 影响面分析。
4. 方案分叉与决策。
5. 契约变更矩阵。
6. 关键设计约束。
7. 不明确项（Design）。
8. 风险记录（Design）。
9. 实现逻辑细化。
10. 人工待补 TODO。
11. Design Gate。
12. 回写主设计要求。

## 6. 分层边界（Design vs Implementation）
1. Design 必须明确：
   - 做什么、怎么算、按什么契约返回。
2. Implementation 负责：
   - 具体代码实现、测试实现、性能调优、发布执行。
3. 禁止在 Implementation 阶段重新定义业务口径。

## 7. Gate（通过条件）
1. 推荐方案已落锤，且不选理由明确。
2. 契约变更矩阵完整。
3. 逻辑细化章节完成，达到可编码输入。
4. `P0/P1` 不明确项关闭。
5. `P2/P3` 仅允许 `accepted-risk + owner + due_at`。
6. TODO 项均有 owner，且有处理结论（完成或带风险放行）。

## 8. 失败回退
1. 需求歧义：回 Requirement 补充澄清。
2. 技术不可行：回 Design 重新分叉。
3. 风险不可控：缩范围重切片后重审。

## 9. 本次迭代沉淀（可复用）
1. 同一功能模块长期只维护一份主设计 `design/design.md`，每次迭代写独立 `design/iterations/<...>/design.md`。
2. 迭代设计中，”全国卡片与评分明细总计同结构”这类口径一致性必须写成硬约束。
3. 数据语义可明确允许 `null` 与 `0` 同时存在，但必须定义语义边界：
   - 无数据 -> `null`
   - 真实计算结果为零 -> `0`
4. 指标类需求优先走配置驱动（如 `/api/target/view`），避免业务侧硬编码扩散。
5. 接口文档规范（已确认）：
   - 主设计 `design.md` 必须包含每个接口的完整字段级契约（请求 + 响应），作为迭代 Design 的基线。
   - 接口文档每个接口独立展示完整字段，不抽公共字段单独罗列，便于直接阅读和联调。
   - 字段信息来源：从源码 DTO/VO + Service 实现中提取，不依赖注释或文档推断。
   - 迭代 Design 输出变更后的完整接口契约，不只写增量 diff。
   - 主设计评审通过后，接口契约基线随主设计一并回写更新。

## 10. 接口文档规范（强制）

### 10.1 接口文档写在哪

| 文档 | 职责 |
|------|------|
| 主设计 `design/design.md` | 现有接口完整字段级契约基线，迭代评审通过后回写更新 |
| 迭代设计 `design/iterations/.../design.md` | 变更后的完整接口契约，不只写 diff，展示变更后全貌 |

### 10.2 格式规范（强制）

1. **每个接口独立展示完整字段**，不抽公共字段单独罗列。
2. 请求字段表格必填列：`字段名 / 类型 / 是否必填 / 说明`。
3. 响应字段说明每个关键字段的类型与业务语义。
4. 内部注入字段必须标注（如 `rankType` 由 controller 注入，`platform` 由 service 推导）。
5. 异步语义接口必须标注”接口成功 ≠ 业务完成”。
6. 字段信息从源码 DTO/VO + Service 实现中提取，不依赖注释推断。

### 10.3 主设计接口契约缺失时的处理

1. 进入 Design 阶段前，先检查主设计是否有接口契约。
2. 若缺失：AI 从源码 DTO + Service 实现提取字段，补充到主设计后再生成迭代设计。
3. 若无源码访问权限：登记 `TODO-DES-*` 并标注 owner，进入 Implementation 前必须完成。
