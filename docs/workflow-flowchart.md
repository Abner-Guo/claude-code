# 知识基座 → Requirement → Design 全流程图

## 文档信息
- 状态: V1.0
- 日期: 2026-04-10
- 来源: 基于实操过程记录 + 阶段规范 + Skill 设计综合整理
- 定位: 明确每个节点的执行主体（AI 自动化 / AI 询问人 / 人工审核）

---

## 节点类型图例

| 标记 | 含义 | 说明 |
|------|------|------|
| `[S]` | **Skill — AI 自动化** | 可封装为 Skill，AI 独立完成，无需人介入 |
| `[Q]` | **AI 主动询问** | AI 执行到此必须向用户提问，拿到答案才继续 |
| `[H]` | **人工审核/决策** | 必须由人在企业内部完成，AI 不可替代 |
| `[G]` | **Gate 检查点** | AI 生成报告 + 人最终放行（双主体） |

## 一、知识基座建设流程

> 对应 Skill：知识包生成 + 主设计反向生成
> 触发时机：新功能首次建基座，或存量系统补基座

```
┌─────────────────────────────────────────────────────────┐
│                  知识基座建设流程                          │
└─────────────────────────────────────────────────────────┘

  输入: controller_path
  │
  ▼
┌─────────────────────────────────────┐
│ 1. [S] 解析 Controller              │
│    提取路由、请求类型、注解、         │
│    service 注入、方法映射             │
└──────────────┬──────────────────────┘
               │
               ▼
┌─────────────────────────────────────┐
│ 2. [S] 解析 Service                  │
│    接口与实现、方法级逻辑、           │
│    关键分支、异步行为                 │
└──────────────┬──────────────────────┘
               │
               ▼
┌─────────────────────────────────────┐
│ 3. [S] 解析 Mapper + XML            │
│    方法映射、SQL 段、表依赖、         │
│    数据源依赖                        │
└──────────────┬──────────────────────┘
               │
               ▼
┌─────────────────────────────────────┐
│ 4. [S] 抽取请求/响应模型             │
│    DTO/VO 字段、类型、业务含义       │
└──────────────┬──────────────────────┘
               │
               ▼
┌─────────────────────────────────────┐
│ 5. [S] 生成完整知识包（8 类交付物）   │
│    controller / service / module /   │
│    feature / relation / table /      │
│    index.md / index.json             │
│    ─────────────────────────────     │
│    命名冲突自动升级                   │
│    所有文档统一 draft 状态            │
│    证据附 文件路径 + 行号             │
└──────────────┬──────────────────────┘
               │
               ▼
┌─────────────────────────────────────┐
│ 6. [Q] 输出审核待办清单              │
│    ● DDL 缺失项（需人工补齐）        │
│    ● 高风险规则复核项                │
│    ● 命名冲突决策项                  │
│    ● 跨数据源/异步语义需确认项       │
│    → AI 主动告知用户需补充哪些内容   │
└──────────────┬──────────────────────┘
               │
               ▼
┌─────────────────────────────────────┐
│ 7. [H] 人工审校知识文档              │
│    ● 框架层 → 架构师/Tech Lead       │
│    ● 模块层 → 模块 owner             │
│    ● Controller → 接口 owner         │
│    ● Table → DBA + 业务 owner        │
│    ─────────────────────────────     │
│    补齐 DDL、修正错误结论、           │
│    标记 verified + verified_by        │
└──────────────┬──────────────────────┘
               │
               ▼
┌─────────────────────────────────────┐
│ 8. [S] 主设计反向生成                │
│    基于知识基座 + 源码，生成          │
│    design/design.md（draft）         │
│    ─────────────────────────────     │
│    含接口契约基线（每接口独立完整）   │
│    标注内部注入字段、异步语义         │
└──────────────┬──────────────────────┘
               │
               ▼
┌─────────────────────────────────────┐
│ 9. [Q] 输出主设计审核待办            │
│    ● 异步语义接口确认                │
│    ● 权限注解异常项                  │
│    ● 高风险字段/口径确认             │
│    → AI 主动列出需人工确认的项       │
└──────────────┬──────────────────────┘
               │
               ▼
┌─────────────────────────────────────┐
│ 10. [H] 人工审校主设计               │
│     接口契约是否准确                  │
│     业务规则是否完整                  │
│     审校通过 → status: verified       │
└──────────────┬──────────────────────┘
               │
               ▼
           【基座就绪】
               │
    ┌──────────┴──────────┐
    ▼                     ▼
  日常迭代进入        持续治理
  (Requirement)       │
                      ▼
              ┌───────────────────┐
              │ [S] PR 漂移检测    │
              │ kb drift           │
              │ 代码变化命中规则   │
              │ → 强制补文档       │
              └───────────────────┘
```

## 二、Requirement 阶段流程

> 对应 Skill：kb 检索 + Requirement Gate Check
> 触发时机：新迭代需求进入

```
┌─────────────────────────────────────────────────────────┐
│                  Requirement 阶段                        │
└─────────────────────────────────────────────────────────┘

  输入: 需求原文（PRD/飞书/wiki/口头）
  │
  ▼
┌─────────────────────────────────────┐
│ 1. [S] 创建迭代需求文档结构          │
│    docs/features/<feature>/          │
│      requirement/iterations/         │
│        <date-topic>/                 │
│          requirement.spec.md         │
└──────────────┬──────────────────────┘
               │
               ▼
┌─────────────────────────────────────┐
│ 2. [Q] 获取需求原文                  │
│    ● 飞书/wiki → 尝试自动拉取       │
│    ● 拉取失败 → 请用户提供原文       │
│    ● 口头需求 → 请用户口述要点       │
└──────────────┬──────────────────────┘
               │
               ▼
┌─────────────────────────────────────┐
│ 3. [S] 两级检索 + 双源校验           │
│    一级: 读历史主设计 design.md      │
│    二级: 读 controller/module/table  │
│          知识基座                     │
│    校验: 文档结论 ↔ 代码证据对齐     │
│          (file + line + commit)      │
└──────────────┬──────────────────────┘
               │
               ▼
┌─────────────────────────────────────┐
│ 4. [S] 生成结构化需求草案            │
│    背景、目标、范围/非范围、          │
│    约束、验收标准                     │
│    （基于检索结果 + 需求原文）        │
└──────────────┬──────────────────────┘
               │
               ▼
┌─────────────────────────────────────┐
│ 5. [Q] 澄清范围与验收               │
│    ● 范围/非范围是否准确？           │
│    ● 验收标准是否可测试？            │
│    ● 是否存在接口契约变更？          │
│    ● 是否影响历史能力？              │
│    → 逐项与用户确认，拿到明确答案    │
└──────────────┬──────────────────────┘
               │
               ▼
┌─────────────────────────────────────┐
│ 6. [S] 冲突与影响识别                │
│    ● 接口契约是否变化                │
│    ● 统计口径是否变化                │
│    ● 是否影响历史能力                │
│    ● 标记"有/无契约变更 + 影响范围"  │
└──────────────┬──────────────────────┘
               │
               ▼
┌─────────────────────────────────────┐
│ 7. [S] 登记不明确项 + 风险           │
│    写入 uncertainty/register.md      │
│    写入 risk/register.md             │
│    每项: id/state/question/impact/   │
│          default_assumption/owner    │
└──────────────┬──────────────────────┘
               │
               ▼
┌─────────────────────────────────────┐
│ 8. [Q] 不明确项逐条决议              │
│    ● P0/P1 项 → 必须当场解决        │
│    ● AI 提出默认假设供用户确认       │
│    ● 用户回答后更新 register 状态    │
│    （answered / accepted-risk /      │
│     rejected）                       │
└──────────────┬──────────────────────┘
               │
               ▼
┌─────────────────────────────────────┐
│ 9. [Q] 确认阶段边界                  │
│    ● 哪些下沉到 Design？            │
│    （字段级契约、SQL 粒度、          │
│     实现分支 → 下沉）                │
│    ● Requirement 只保留：            │
│     有无变更 + 影响范围 + 兼容方向   │
└──────────────┬──────────────────────┘
               │
               ▼
┌─────────────────────────────────────┐
│ 10. [S] 回填需求文档                 │
│     将上述所有结论写入               │
│     requirement.spec.md              │
└──────────────┬──────────────────────┘
               │
               ▼
┌─────────────────────────────────────────────────┐
│ 11. [G] Requirement Gate Check                   │
│     ─────────────────────────────────────────    │
│     [S] AI 自动校验（5 条）:                     │
│       ① 目标/范围/非范围明确                     │
│       ② 验收标准可测试                           │
│       ③ 契约变更已确认                           │
│       ④ P0/P1 不明确项关闭                       │
│       ⑤ P2/P3 已 accepted-risk + owner + due_at  │
│     ─────────────────────────────────────────    │
│     → 输出 Gate 报告（PASS / FAIL）              │
│     → 追加到 requirement.spec.md 末尾            │
│                                                  │
│     FAIL → 列出必须解决项，回退到步骤 5/8 补充   │
│     PASS ↓                                       │
│     ─────────────────────────────────────────    │
│     [H] 人工最终放行                             │
│       确认进入 Design / 回退补充                  │
│       放行后 status → approved                    │
└──────────────┬──────────────────────────────────┘
               │ PASS + 人工放行
               ▼
         【进入 Design】
```

## 三、Design 阶段流程

> 对应 Skill：主设计反向生成（补基线时）+ Design Gate Check
> 触发时机：Requirement Gate 通过且人工放行后

```
┌─────────────────────────────────────────────────────────┐
│                    Design 阶段                           │
└─────────────────────────────────────────────────────────┘

  输入: approved requirement.spec.md
      + 历史主设计 + 知识基座 + 寄存文档
  │
  ▼
┌─────────────────────────────────────┐
│ 1. [S] 检查主设计接口契约基线        │
│    读取 design/design.md             │
│    → 有完整接口契约？                │
│      YES → 步骤 3                    │
│      NO  ↓                           │
└──────────────┬──────────────────────┘
               │ 缺失
               ▼
┌─────────────────────────────────────┐
│ 2. [S] 补充接口契约基线              │
│    从源码 DTO/VO + Service 实现      │
│    提取字段，写入主设计              │
│    （等同于主设计反向生成 Skill）     │
│    ─────────────────────────────     │
│    无源码访问 → 登记 TODO-DES-*      │
│    并通知用户补充                     │
└──────────────┬──────────────────────┘
               │
               ▼
┌─────────────────────────────────────┐
│ 3. [S] 创建迭代设计文档结构          │
│    docs/features/<feature>/          │
│      design/iterations/              │
│        <date-topic>/design.md        │
└──────────────┬──────────────────────┘
               │
               ▼
┌─────────────────────────────────────┐
│ 4. [S] 影响面建模                    │
│    识别影响的 controller / service / │
│    mapper / table 与兼容面           │
│    输出 impact.analysis              │
└──────────────┬──────────────────────┘
               │
               ▼
┌─────────────────────────────────────┐
│ 5. [S] 方案分叉（至少 2 个方案）     │
│    每个方案包含:                      │
│    技术路线、优劣、风险、工作量       │
└──────────────┬──────────────────────┘
               │
               ▼
┌─────────────────────────────────────┐
│ 6. [Q] 方案收敛 — 请用户决策         │
│    ● 展示 ≥2 个方案 + 推荐理由      │
│    ● 明确不选方案的不选理由          │
│    ● 等待用户拍板推荐方案            │
└──────────────┬──────────────────────┘
               │ 用户选定方案
               ▼
┌─────────────────────────────────────┐
│ 7. [S] 契约变更矩阵输出             │
│    每个变更接口独立展示完整           │
│    请求/响应字段（不抽公共）         │
│    标注内部注入字段、异步语义         │
└──────────────┬──────────────────────┘
               │
               ▼
┌─────────────────────────────────────┐
│ 8. [S] 可编码级逻辑细化              │
│    ● 请求入参归一化规则              │
│    ● 场景分流规则                    │
│    ● 查询过滤与聚合口径              │
│    ● 返回结构与 null/0 语义          │
└──────────────┬──────────────────────┘
               │
               ▼
┌─────────────────────────────────────┐
│ 9. [Q] 业务口径与实现对齐            │
│    ● 字段名是否与现有代码一致？      │
│    ● SQL 逻辑是否对齐现有写法？      │
│    ● 边界语义确认（null vs 0）       │
│    ● 过滤条件叠加 vs 互斥？          │
│    → 需要读代码验证 + 用户确认       │
│    （实操中这一步往往需要多轮）       │
└──────────────┬──────────────────────┘
               │
               ▼
┌─────────────────────────────────────┐
│ 10. [S] 生成实施切片                 │
│     拆分为可独立交付、               │
│     可独立验证的切片                 │
│     输出 implementation.slices.md    │
└──────────────┬──────────────────────┘
               │
               ▼
┌─────────────────────────────────────┐
│ 11. [S] 登记不明确项 + 风险          │
│     更新 uncertainty/register.md     │
│     更新 risk/register.md            │
└──────────────┬──────────────────────┘
               │
               ▼
┌─────────────────────────────────────┐
│ 12. [Q] 不明确项逐条决议             │
│     ● P0/P1 必须解决                 │
│     ● 提出假设 → 用户确认           │
│     ● 需要读代码反查的 → AI 先查    │
│       再呈现结论请用户确认           │
│     （实操: U-DES-104 指标阈值       │
│      就是靠读代码纠正了错误假设）    │
└──────────────┬──────────────────────┘
               │
               ▼
┌─────────────────────────────────────┐
│ 13. [S] 登记 TODO-DES-* 项          │
│     需人工补充的信息                  │
│     每条绑定 owner                   │
└──────────────┬──────────────────────┘
               │
               ▼
┌─────────────────────────────────────┐
│ 14. [Q] TODO 项 owner 确认           │
│     ● 每个 TODO 是否有 owner？       │
│     ● 处理结论：完成 or 带风险放行？ │
│     → 请用户分配并确认               │
└──────────────┬──────────────────────┘
               │
               ▼
┌─────────────────────────────────────────────────┐
│ 15. [G] Design Gate Check                        │
│     ─────────────────────────────────────────    │
│     [S] AI 自动校验（6 条）:                     │
│       ① 推荐方案已落锤 + 不选理由明确           │
│       ② 契约变更矩阵完整                        │
│       ③ 逻辑细化达到可编码输入                   │
│       ④ P0/P1 不明确项关闭                       │
│       ⑤ P2/P3 已 accepted-risk + owner + due_at  │
│       ⑥ TODO 项均有 owner + 处理结论             │
│     ─────────────────────────────────────────    │
│     → 输出 Gate 报告（PASS / FAIL）              │
│     → 追加到 design.md 末尾                      │
│                                                  │
│     FAIL → 列出必须解决项，回退到对应步骤补充    │
│     PASS ↓                                       │
│     ─────────────────────────────────────────    │
│     [H] 人工最终放行                             │
│       确认进入 Implementation / 回退补充          │
│       放行后 status → approved                    │
└──────────────┬──────────────────────────────────┘
               │ PASS + 人工放行
               ▼
┌─────────────────────────────────────┐
│ 16. [S] 回写主设计                   │
│     将本次设计结论合并到             │
│     design/design.md                 │
│     同步更新接口契约基线             │
└──────────────┬──────────────────────┘
               │
               ▼
┌─────────────────────────────────────┐
│ 17. [H] 确认主设计更新               │
│     回写内容是否准确                  │
│     接口契约基线是否完整              │
└──────────────┬──────────────────────┘
               │
               ▼
         【进入 Implementation】
```

## 四、节点分类汇总

### 4.1 可封装为 Skill 的节点（AI 自动化）

| 阶段 | 节点 | 对应 Skill | 说明 |
|------|------|------------|------|
| 知识基座 | 1~5 | 知识包生成 | 从 controller 路径出发，自动解析全链路并生成 8 类文档 |
| 知识基座 | 8 | 主设计反向生成 | 从代码反向生成 design.md，含接口契约基线 |
| 知识基座 | 持续 | kb drift | PR 触发漂移检测，命中规则强制补文档 |
| Requirement | 1 | 文档结构初始化 | 自动创建目录和文件模板 |
| Requirement | 3 | kb search/read/context | 两级检索 + 双源校验 |
| Requirement | 4 | 需求草案生成 | 基于检索结果 + 需求原文生成结构化草案 |
| Requirement | 6 | 冲突识别 | 自动比对历史，识别变更点 |
| Requirement | 7, 10 | 寄存登记/文档回填 | 自动写入 register 和 spec 文档 |
| Requirement | 11(前半) | Requirement Gate Check | 自动校验 5 条 Gate 条件，输出报告 |
| Design | 1~2 | 接口契约基线检查/补充 | 检查主设计，缺失时从源码提取 |
| Design | 3 | 文档结构初始化 | 自动创建迭代设计目录和文件 |
| Design | 4 | 影响面分析 | 自动识别影响范围 |
| Design | 5 | 方案分叉 | 自动生成 ≥2 方案 |
| Design | 7 | 契约矩阵 | 自动输出变更后完整契约 |
| Design | 8 | 逻辑细化 | 自动生成可编码级规则描述 |
| Design | 10 | 切片生成 | 自动拆分实施切片 |
| Design | 11, 13 | 寄存登记/TODO登记 | 自动写入 register 和 TODO |
| Design | 15(前半) | Design Gate Check | 自动校验 6 条 Gate 条件，输出报告 |
| Design | 16 | 主设计回写 | 自动将迭代结论合并到主设计 |

### 4.2 AI 必须主动询问用户的节点

| 阶段 | 节点 | 询问内容 | 为什么不能自动化 |
|------|------|----------|------------------|
| 知识基座 | 6 | DDL 缺失、高风险项、命名冲突 | DDL 在数据库侧，AI 无权限；命名冲突涉及团队约定 |
| 知识基座 | 9 | 异步语义、权限异常、高风险字段 | 业务语义需人确认，AI 只能标注疑点 |
| Requirement | 2 | 获取需求原文 | 外部系统内容 AI 不一定能自动拉取 |
| Requirement | 5 | 范围/非范围、验收标准、契约变更 | 业务决策，AI 给草案但人拍板 |
| Requirement | 8 | 不明确项逐条决议 | P0/P1 涉及业务口径或技术边界，AI 无法独立判断 |
| Requirement | 9 | 阶段边界确认 | 哪些下沉 Design 取决于当前迭代复杂度，人定 |
| Design | 6 | 方案选择 | 架构/技术方向决策，必须人拍板 |
| Design | 9 | 业务口径与实现对齐 | 字段名、SQL 写法、边界语义需多轮确认 |
| Design | 12 | 不明确项逐条决议 | 同 Requirement 阶段，业务判断不可自动化 |
| Design | 14 | TODO owner 分配 | 涉及团队分工，AI 不知道谁负责什么 |

## 五、关键设计决策说明

### 5.1 为什么 Gate 是双主体（AI + 人）

| 角度 | 说明 |
|------|------|
| AI 不能自动放行 | P0/P1 不明确项通常涉及业务口径或技术边界，AI 无法独立判断是否真的"关闭" |
| AI 不能省略 | 人工逐条检查容易遗漏，AI 做结构化核查确保不漏项 |
| 正确分工 | AI 负责"核查完整性"，人负责"决策是否放行" |

### 5.2 为什么 [Q] 节点不能合并到 [S]

实操经验（评价汇总案例）证明：
1. **U-DES-104**：AI 默认假设"阈值由配置系统驱动"，实际是 SQL 硬编码。如果不主动询问，会带着错误假设进入 Implementation。
2. **SQL 字段名**：AI 推测 `overall_experience_score_latest`，实际是 `overall_experience_score`。不问就错。
3. **过滤条件互斥**：AI 设计为互斥 `if/else if`，实际叠加更简单。不确认就多做了防御性约束。

结论：AI 能高效生成草案，但业务口径和实现细节必须通过询问校准。

### 5.3 哪些 [Q] 节点未来可能升级为 [S]

| 节点 | 升级条件 | 说明 |
|------|----------|------|
| Requirement-2 获取需求原文 | 飞书/wiki API 稳定接入 | 目前已有 feishu-doc-integration，但不是所有需求都能自动拉取 |
| Requirement-6 冲突识别 | 知识基座覆盖��足够高 | 基座不全时，AI 识别的变更可能不完整 |
| Design-9 实现对齐 | 代码规范高度统一 + 基座完整 | 如果 SQL 写法和命名规范足够一致，部分对齐可自动完成 |

## 六、Skill 实现优先级建议

基于流程依赖关系和收益，推荐实现顺序：

| 优先级 | Skill | 理由 |
|--------|-------|------|
| P0 | 知识包生成 | 基座是所有后续流程的输入，没有基座其他 Skill 效果打折 |
| P0 | Requirement Gate Check | 门禁是流程纪律的核心，优先固化避免人工遗漏 |
| P1 | 主设计反向生成 | 解决存量功能无主设计的问题，阻塞迭代 Design 输入 |
| P1 | Design Gate Check | 与 Requirement Gate 同模式，低成本扩展 |
| P2 | kb search/read/context | 检索体系，提升 Requirement 阶段效率 |
| P2 | kb drift | 持续治理，防止基座过期 |
